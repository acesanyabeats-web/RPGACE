====================================================================
WHY THIS HAS BEEN SLOW — real reasons, not excuses
====================================================================
1. Bookworm was built from a cold idea to a full working pipeline (chapter
   detection, per-insight placement cascade, Council-of-5 scoring, streaming
   analysis, 3 entry points) in one continuous session. That's a big build,
   and every real hand-test against a real book surfaced a genuinely new bug
   — "Chapter 0 of 0", TOC-as-chapter-1 (twice), 87 fake chapters, cumulative
   repeated taxonomy paths, a 7-minute wait before the first insight, and now
   the 19-of-26-chapters bug. Each one was real and needed fixing; the volume
   of them in one sitting is what made it feel endless.
2. One diagnostic mismatch cost several rounds: for a while, fixes were
   landing correctly in the manual-TOC-paste flow while Alex kept re-testing
   through the original "📖 Start" URL button with a different URL each time
   — a different code path entirely. Neither side noticed for a few rounds.
   Querying Supabase directly (source_url, chapter counts) is what finally
   caught it — that's now a standing rule in CLAUDE.md.
3. Long conversations lose fidelity. By the time this session got long, both
   sides were re-explaining context that should have been durable (this is
   exactly why bookworm_spec_backlog.txt and this file exist — chat memory
   is not durable storage).

HOW TO MAKE THE NEXT SESSION FASTER — concrete, for Alex
====================================================================
1. When reporting a bug, say EXACTLY which button/entry point you used and
   paste the exact URL or text you typed, not just the resulting screen.
   ("I clicked 📖 Start and pasted <url>" vs "I pasted the TOC" are different
   code paths in Bookworm — the July 17 confusion was entirely this.)
2. One bug at a time. If a fix doesn't work, say so and stop there — don't
   layer a second new ask (copyright link, new UI merge request, etc.) on
   top of an unresolved bug in the same breath. Each layered ask restarts
   context-gathering.
3. Let a diagnosis finish before redirecting. "Query Supabase directly" is
   now a standing rule precisely because guessing from a symptom description
   alone burned multiple rounds this session.
4. Big multi-part asks (3+ new pieces, like the Function1/Function2 merge)
   go through a spec pass first — that's already a project rule, and it's
   the rule that saved the most confusion this session, not the one that
   caused it.
5. Say "update oversight" / "end session" explicitly when you're done for
   the day — that's still the only reliable trigger (no auto-hook exists).

TIER 2 — known architecture bugs, not Bookworm-specific
5. Oracle 504 timeout on long responses. Root cause confirmed: callClaude()
   is one blocking non-streaming call. A prior streaming attempt was
   reverted — dead code + restoreSendChat neutralizer still present in
   rpgace_core.js. Needs a deliberate streaming/chunking redesign, not
   another token trim (the current workaround pattern across the whole
   codebase — 700 tokens works, 1200 truncates, 1800+ fails outright — is a
   band-aid, not a fix).
6. ~25 unaudited occurrences of `hooks.on('rpgace:ready', ...)` in
   rpgace_core.js. The bug shape (confirmed twice already in phylumPath):
   any module subscribing to 'rpgace:ready' from inside its OWN init() is
   too late to ever fire, silently, forever. Needs: grep every occurrence,
   check whether it's inside that module's own init() (buggy) or genuinely
   elsewhere (fine), fix each buggy one by calling the function directly.
7. `_generateNodeContent`'s empty deep_content mystery — partially resolved
   in a prior session, never fully verified fixed.
