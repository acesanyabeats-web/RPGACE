============================================================
ORIGINAL 4 QUESTIONS ASKED
============================================================
1. Ingestion method — given the 8000-char fetch cap in fetchURL()
   (api/_context.js) makes one-shot whole-book URL ingestion unreliable,
   how should Bookworm actually get chapter text in?
2. Phylum scope — does a book get one fixed phylum, or auto-detect per
   chapter/insight?
3. Confirm granularity — whole chapter at once, or one insight at a time?
4. Bibliography home — new dedicated table, or fold into Fons Educationis
   (Phylum 12)?

============================================================
ANSWERS
============================================================

Q1 — FINAL, CORRECTED AGAIN (verbatim, the version actually being built):
"Jina fetches only book contents first, names of chapters, that loads
as steps for progress bar, then the progress bar sends kina what
chapter to extract, one after the other. chapter 1 gets called, goes
through function, all insights loaded down whole system, then approved.
then chapter 1 is done, move to chapter 2. kina is then told chapter 2
title, kina gets text under chapter 2 title, whole cycle again. repeat
cycle till chapters left to study is 0."
Interpretation, and the exact implementation being built: paste the
book's URL once. ONE full uncapped Jina fetch happens up front (new
dedicated fetch, NOT the existing capped fetchURL() in api/_context.js
— that 8000-char cap stays untouched for its other callers). From that
one fetch, chapter boundaries are detected (regex/heuristic pass for
common chapter-heading patterns, Oracle fallback if that doesn't find
enough) and EVERY chapter's text is sliced out immediately in that same
pass — this is what makes the total chapter count (and therefore the
progress bar) known right away, and avoids re-fetching Jina per chapter
(slower, more failure-prone, and unnecessary once the full text is
already in hand). "Jina extracts chapter N" is implemented as a local
slice against the already-fetched text, not a fresh network call each
time — functionally identical to what's described, just avoids hitting
Jina N times for one book.
REAL TECHNICAL RISK, flagged honestly rather than promised to just
work: Jina Reader reliably returning the FULL text of a book-length PDF
in one response, within Vercel's function time/payload limits, is
unverified — large PDFs are exactly the class of thing that already
caused the original ingestion error this whole feature grew out of.
Plan: attempt the full fetch; if it fails/times out, surface a clear
error rather than silently truncating or faking progress data.

Q2 — USER'S ACTUAL ANSWER (verbatim — I wrongly substituted my own
recommended default the first time around, this is the real one):
"most closely related phylum, add each insight down taxonomy tree after
I approve."
Interpretation: NOT one fixed phylum chosen upfront for the whole book.
Each insight gets matched to whichever phylum is most closely related
(auto-detected, same scoring concept as the existing auto-detect badge),
and once approved, gets inserted into THAT phylum's tree. Overrides the
earlier "one phylum per book" default entirely.

Q3 — SUPERSEDED, same message, verbatim:
"To approve, I want the pop up to first show the chapter for me to read,
once I click read, the insight summary is shown for me, then it's
taxonomy path, then justification for path. When approved, the leaf is
made, if rejected, the next insight is loaded with summary, path and
path justification. If edit, give oracle chat where I give my own
version of path. Do this all continuously till no insights for [chosen]
phylum left."
Interpretation: NOT a whole-chapter batch confirm (my original
recommended default was also wrong here). It's PER-INSIGHT, sequential:
read the chapter ONCE per chapter (a single Read button/popup), then
cycle through each insight one at a time — insight summary, then
taxonomy path, then justification for that path, then:
  - Approve -> the leaf is created in taxonomy_tree
  - Reject -> advance to the next insight (summary/path/justification)
  - Edit -> open an Oracle chat turn where the user gives their own
    version of the path
Repeats continuously until every insight for the chosen/matched phylum
is exhausted.

NEW — phylum fallback cascade (volunteered this message, not one of the
original 4 questions, verbatim):
"then any insights that not for chosen phylum then get put against
other phylums, and goes through same read summary path and justification.
then insights that have not found a home then go through oracle to come
up with where the insight should belong."
Interpretation: after exhausting insights that fit the primary/most-
related phylum, any leftover insights that didn't fit get checked
against the OTHER phyla, one at a time, through the SAME read -> summary
-> path -> justification -> approve/reject/edit loop. Any insight that
still doesn't fit anywhere after that gets a final dedicated Oracle
placement-reasoning pass to determine a home from scratch.

NEW — Council of 5 / confidence-scoring loop (verbatim):
"it should also use /Godmode and council of 5, where 5/10 insights get
rephrased till unanimous 9/10, anything under 4 can be seen if possible
to upgrade at all for leaf."
Interpretation (MY BEST-FAITH READ, NOT YET CONFIRMED — flagging this
explicitly rather than silently assuming, same mistake as Q2/Q3 the
first time): each proposed placement gets scored via a Council-of-5-
style multi-perspective confidence check, expressed out of 10.
  - Score 9-10 -> proceed to show the user (read/summary/path/
    justification/approve-reject-edit as above)
  - Score 5-8 -> Oracle rephrases the insight's wording and re-scores,
    looping until 9-10 is reached (capped at a small number of attempts
    to avoid an infinite loop — exact cap not specified by the user yet)
  - Score under 4 -> a distinct, lighter check: "is this even upgradeable
    to leaf-quality at all" — if not, it doesn't get forced into a leaf,
    it's flagged as unplaced instead
Planned implementation: reuse the same 5-check framework already in
phylumPath.decidePlacement() (pedagogical clarity / non-redundancy /
practical applicability / structural fit / expansion headroom), each
worth up to 2 points, scored by one extra Oracle self-critique call
after a candidate placement is produced. NOT yet confirmed: exact
scoring mechanics (is this literally 5 separate model votes summed, or
one call producing a 1-10 confidence number directly), or the rephrase-
attempt cap. Will implement with a reasonable, clearly-commented
interpretation and flag it for correction rather than blocking on it.

Q4 — ANSWERED (verbatim, given alongside the Q1 correction above):
"then book gets loaded into bibliography section in research."
Interpretation: neither of the two original options exactly — not a
brand new standalone Dashboard module, and not folded into the
taxonomy_tree as a Fons Educationis node. A dedicated `bibliography`
table backs the data (title, source_url, total chapters/insights,
phyla touched, completed_at), but the UI section that LISTS completed
books lives inside the EXISTING Research page/tab (`#page-research`,
same page phylumPath's own manual-entry button already anchors to),
not as its own new top-level Dashboard tab.

Q1 — ANSWERED: clicking an already-COMPLETE chapter in the new
clickable list — what should happen?
"Read-only insight summary" — opens a view showing that chapter's
insights and what was decided (approved/rejected/edited) for each;
nothing re-runs, nothing can be re-approved. Built as
_renderChapterSummary().

Q2 — ANSWERED: when you click "I've Read This — Show Insights", should
it wait to show insight 1, or go straight to background + Dashboard?
Verbatim: "same waiting pop-up when insight approved, denied, but with
exit button, when extracting chapters go straight to dashboard" —
interpreted as: the per-insight approve/reject review popup keeps its
existing wait-based behavior unchanged (still shows insight 1 as soon
as it's ready, still has the "waiting for next insight" polling
screen for insights 2+) — but the CHAPTER EXTRACTION step itself
(clicking "I've Read This" to kick off _analyzeChapter) no longer
waits even for insight 1: it closes the overlay and navigates to
Dashboard immediately, with analysis continuing fully in the
background (it already ran in the background past insight 1 before
this change — see _continueAnalyzingInBackground, built July 17 - this
extends that same principle to insight 1 as well).

BUILT this round (rpgace_core.js, bookworm module):
- _goToDashboard() — shared helper, every Bookworm overlay's "Exit"
  button now navigates to the Dashboard page (main.js's showPage())
  instead of just removing the overlay and leaving whatever page was
  underneath.
- _openBook(bookId) — re-entry point (Dashboard widget's book-card
  click) now shows a full chapter list (_renderChapterList) instead of
  jumping straight to the current checkpoint. Every chapter gets a
  status tick reusing the EXISTING bookworm_chapters.status field
  (pending/in_progress/complete) plus raw_text/insights presence -
  no new columns needed. Click any chapter to jump straight there.
- _openCurrentChapter(bookId) — the OLD _openBook logic, kept
  separately for internal "keep going" flows that should still jump
  straight to the current checkpoint without showing the list every
  time: starting a fresh book, saving a TOC-manual chapter's pasted
  text, and auto-advancing after a chapter completes.
- _renderChapterSummary(book, chapter) — read-only view for a
  complete chapter's insights and their final decisions, per Q1.
- _renderChapterRead's "I've Read This" button — per Q2, now
  fire-and-forget: toasts, closes the overlay, navigates to Dashboard,
  and lets _analyzeChapter run without the UI awaiting it. Errors
  still surface as a toast (not tied to the overlay's lifetime).

Not yet hand-tested — flag honestly, same as every other Bookworm

ROUND 3 — July 19: hardening background analysis against a closed tab
+ background leaf-creation queue for Approve
============================================================
Two direct requests in one message, both about background-job
durability:
1. Confirmed as a real gap in Round 2's answer ("say the word if you
   want that hardened") — "Yes please."
2. "when clicked green button, go straight to next insight for
   approval, let create leaf go down a pipeline with a queue in
   background and creation in background to continue working"

New Supabase columns (migration applied directly):
bookworm_chapters.pending_insight_texts (jsonb, default '[]') and
bookworm_chapters.analysis_heartbeat (timestamptz). No new column
needed for per-insight leaf status - insights is already jsonb, so
each insight object just gets a leafStatus field ('pending'/'created'/
'failed').

BUILT this round (rpgace_core.js, bookworm module):
- _analyzeChapter now persists the FULL remaining insight-text list
  (pending_insight_texts) and the resolved primaryPhylum
  (suggested_phylum) up front, before placing even the first insight -
  this is the durable checkpoint a resume reads back.
- _continueAnalyzingInBackground drops each insight's text from
  pending_insight_texts as it's placed, and stamps analysis_heartbeat
  after every step.
- _resumeChapterAnalysis(book, chapter) - reads pending_insight_texts +
  suggested_phylum FRESH from Supabase (never JS memory) and re-enters
  _continueAnalyzingInBackground. Same function now serves both live
  continuation and resume from a totally different tab/session.
- _renderWaitingForNextInsight now shows a "▶ Resume Analysis" button
  once analysis_heartbeat has gone stale for 45+ seconds with no
  progress - hidden while things are still genuinely running.
- Honest, NOT solved: no server-side lock exists. If the original tab
  is actually still alive and just slow (not truly stalled), clicking
  Resume could run two chains over the same remaining insights at
  once. The heartbeat threshold makes this unlikely in practice but
  this is a single-user personal tool, not a distributed system - true
  mutual exclusion wasn't built, and would need a real backend job
  queue to do properly.
- approveBtn.onclick now calls advance('approved', {leafStatus:
  'pending'}) IMMEDIATELY (moves to the next insight without waiting),
  then _queueLeafCreation(chapterId, idx, insight) - the actual
  _insertNewSteps call, chained on a single shared queue (_leafQueue)
  so concurrent approvals never run the chained parent_id insert
  against itself. On success/failure the insight's leafStatus gets
  patched to 'created'/'failed' via _patchChapterInsightAt.
- _patchChapterInsightAt(chapterId, idx, patchFn, extraFields) - safe
  read-modify-write for one insight, itself serialized through a
  second shared chain (_chapterWriteQueue). Needed because advance()'s
  own write and the later leaf-creation write both touch the same
  insights array asynchronously - without shared serialization, the
  later one could read a stale array and clobber the earlier one.
  advance() itself now goes through this too, not a blind overwrite.
- _renderChapterSummary (the read-only complete-chapter view) now
  shows "(leaf still writing...)" / "(⚠️ leaf write failed)" next to
  any approved/edited insight whose background write hasn't landed
  cleanly yet, plus a "🔁 Retry Leaf Creation" button on failed ones -
  per this project's "fail loud, never silently swallow a failed
  write" rule, a complete chapter's summary shouldn't imply every
  approved insight definitely made it into taxonomy_tree if one
  didn't.

Not yet hand-tested.
