🐢 July 22 — Real root cause found for "left nav doesn't respond for ~10s after login": onReady() was waiting for the wrong signal, not just for leftNav specifically
Alex's report: page loads, dashboard scrolls fine, but clicking anything in the left list does nothing for the first ~10 seconds. GODMODE evidence pass instead of another guess.

Real root cause: onReady() — the function every one of ~40 modules' init() calls wait behind via rpgace:ready — gated on document.readyState==='complete' (the window 'load' event), which waits for EVERY subresource on the page (every image, every font) to finish downloading, not just the DOM. The comment justifying this said "give initApp() time to run" — checked checkPassword() in main.js directly: initApp() is called from the login handler, gated entirely by WHEN Alex logs in, completely decoupled from page-load timing. It can run long after window.load already fired, or need to run before it ever will. There was never a real dependency on waiting for images/fonts — just unnecessary latency stacked in front of every module's startup, worst-case as long as the slowest image/font on the page takes.

Fix: onReady() now checks document.readyState !== 'loading' (DOMContentLoaded-equivalent — fires once the DOM itself is parsed, which it already is by the time rpgace_core.js runs, since both scripts load at the very end of <body>) instead of waiting for full 'complete', cutting the delay from window.load + 150ms to effectively DOMContentLoaded + 50ms. This is a systemic fix — every module's startup timing improves, not just leftNav's.

Second, separate bug found in the same pass: leftNav._go(page) unconditionally dereferenced RPGACE.CONFIG.pages.phylumPath on every single click, regardless of which page was clicked. If RPGACE.CONFIG ever wasn't ready yet at click time, this threw before ever calling showPage() — the click would silently do nothing, no error shown, no toast, exactly matching Alex's description. Guarded defensively, same pattern as _items()'s existing CONFIG guard.

Stale cache-busting version found again — the version string was bumped once already today (?v=20260722a) after the earlier incident, but rpgace_core.js changed again after that bump for this fix, so it was bumped a second time to ?v=20260722b. Repeating the lesson from earlier today: bump this every time rpgace_core.js changes, not just once per day.

node --check rpgace_core.js clean. Merged to main and pushed before this card was written, per the standing rule. Not yet hand-tested — needs a real hard-refresh + login + immediate click-through from Alex to confirm the lag is actually gone.

🏗️ July 18 — Bookworm's URL/PDF chapter detection rebuilt from scratch, per Alex's own architectural insight
After 4 separate fix attempts on the same missing-chapters bug (whitespace-fuzzy matching, gap-fill retry, bounded structural fallback) each improved things but the missing/misordered set kept shifting between runs — Alex diagnosed the real underlying pattern directly: the manual TOC-paste path has never failed once, across every single test, because it only ever asks the model to read a clean table of contents — it never asks the model to recall body text it may not have actually seen. The URL/PDF path's searchString mechanism did exactly that: asked the model to "quote verbatim" a chapter's opening line, which — confirmed directly via Vercel logs — it sometimes couldn't do, because that chapter's real content lay beyond its 25000-char viewing window (this book's own detailed table of contents alone fills that window). It would then reconstruct a plausible-sounding guess from that chapter's own TOC sub-headings, which correctly failed to string-match anywhere real, because it was never real.

Rebuilt api/bookworm-fetch.js around Alex's proposed architecture — two clean, independent steps instead of one fragile one:
1. detectChapterListByOracle() — the model's ONLY job is reading the table of contents: chapter number, title, page number if shown, keywords, suggested phylum. It is never asked to recall or quote any body text at all — the same scope the manual-paste path has always used, and the same reason that path has never once failed.
2. resolveChapterHeadingsMechanically() — fully deterministic, zero model involvement. Every chapter's number+title pattern is searched for across the WHOLE extracted text; the tightly-clustered front-matter TOC mentions are stripped (same declustering principle as the existing dropClusteredBoundaries); then chapters are walked in reading order, each one assigned the first remaining occurrence after the previous chapter's position. This naturally lands on the real heading, skips the TOC decoy, and skips any later cross-reference (a chapter is essentially never referenced by number before it's actually reached in a book) — solving the ordering bug from the last fix pass as a side effect of the design, not a bolt-on.

Net effect: the URL/PDF path's chapter detection now works on exactly the same principle that's made the manual path 100% reliable every time it's been tested — read a list, don't recall unread prose. The gap-fill retry mechanism (2 prior sessions' worth of patches) is removed entirely; it's structurally unnecessary now, since detection no longer depends on the model having "read" anything beyond the TOC.

Root cause found after both of the above still didn't fully fix it: confirmed via a local Playwright harness (production network is blocked from this environment, so verified against an identical local static serve of the exact deployed code instead) that RPGACE.register(name, module) wires every module's init() as a listener ON 'rpgace:ready' itself — module.init() only ever runs BECAUSE that event already fired. phylumPath.init() was re-subscribing to that SAME event from inside itself, twice (once for the original side-panel button, once for the new Phase 2 nav tab) — but RPGACE.hooks.fire() iterates listeners via a plain Array.forEach, which per spec never revisits entries pushed onto the array after iteration starts. ⚡ July 14 — efficiency audit: 1 real correctness bug + 1 self-inflicted duplication, both fixed
Asked to audit the codebase for efficiency/speed/workload upgrades after Phylum Path shipped. Found two, both fixed (Council of 5 verdict on each: 9/10 — concrete, low-risk, verified before reporting):

1. Cache-busting on write has never actually worked. RPGACE.cache (the 60s in-memory layer transparently wrapping every RPGACE.sb.select() call) stores entries under composite keys (table + '|' + params), but RPGACE.sb.insert()/update()/del()'s cache-busting wrappers all call RPGACE.cache.clear(table) with just the bare table name — which never matched a real stored key, so delete this._store[key] was silently deleting nothing. 2. Self-inflicted duplicate MutationObserver. Phylum Path's own auto-detect (shipped minutes earlier this same session) installed a second observer on #send-btn's disabled attribute, duplicating RPGACE.utils._initPhylaObserver's existing one and re-querying the chat DOM a second time on every single completed Oracle response — exactly the kind of duplicate-implementation pattern this project has repeatedly found and fixed before (three independent phyla-keyword lists, three chat-parsers).
