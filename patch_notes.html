🔧 July 16 — Phase 2 nav tab: mobile CSS + cache-busting fixed, then the real root cause found
Reports that the new "🧬 Phylum Path" nav tab (shipped July 15/16 as Phase 2) simply wasn't appearing at all, even in a fresh incognito load — ruling out ordinary caching as the sole explanation. Two real bugs found and fixed in sequence:

1. No cache-busting on the script tags at all. <script src="rpgace_core.js"></script> had no version query string whatsoever — a phone browser could silently serve an indefinitely stale pre-Phase-2 copy. Added ?v=20260716a to both script tags.

2. Mobile nav CSS was contradictory. .nav-tab { flex:1; min-width:50px } inside a horizontally-scrollable overflow-x:auto container squeezed every tab (including "AI ADVISOR", "ENCYCLOPEDIA") into an equal tiny slice instead of natural readable width, causing the reported overlap/clipping — worse with a 9th tab now in the row. Fixed by removing flex:1, bumping padding/font-size, and adding dynamically-injected left/right scroll-arrow buttons for touchscreens where drag-to-scroll alone was unreliable.

Root cause found after both of the above still didn't fully fix it: confirmed via a local Playwright harness (production network is blocked from this environment, so verified against an identical local static serve of the exact deployed code instead) that RPGACE.register(name, module) wires every module's init() as a listener ON 'rpgace:ready' itself — module.init() only ever runs BECAUSE that event already fired. phylumPath.init() was re-subscribing to that SAME event from inside itself, twice (once for the original side-panel button, once for the new Phase 2 nav tab) — but RPGACE.hooks.fire() iterates listeners via a plain Array.forEach, which per spec never revisits entries pushed onto the array after iteration starts. A listener registered from inside init() can silently never fire on the pass that's currently invoking init() — no error, no retry, nothing. Fixed by calling the injector functions directly instead of re-subscribing. Confirmed fixed: tab now appears on a normal first load in the local harness, no manual re-fire needed.

Flagged, not fixed: grep found ~25 more occurrences of hooks.on('rpgace:ready', ...) in rpgace_core.js beyond R.register's own definition — meaning this exact same latent bug likely affects other modules too, each silently depending on lucky timing rather than working by design. Not audited this pass; logged in /root/.claude/plans/woolly-watching-lamport.md as a real, standing architectural fragility worth a dedicated future pass.

⚡ July 14 — efficiency audit: 1 real correctness bug + 1 self-inflicted duplication, both fixed
Asked to audit the codebase for efficiency/speed/workload upgrades after Phylum Path shipped. Found two, both fixed (Council of 5 verdict on each: 9/10 — concrete, low-risk, verified before reporting):

1. Cache-busting on write has never actually worked. RPGACE.cache (the 60s in-memory layer transparently wrapping every RPGACE.sb.select() call) stores entries under composite keys (table + '|' + params), but RPGACE.sb.insert()/update()/del()'s cache-busting wrappers all call RPGACE.cache.clear(table) with just the bare table name — which never matched a real stored key, so delete this._store[key] was silently deleting nothing. Confirmed via a standalone simulation before and after the fix. Real impact: any re-fetch of a cached table (taxonomy_tree, taxonomy_nodes, etc. — everything except the explicit noCache list) within 60 seconds of a write could silently show stale data — e.g. Phylum Path's tree view not showing a just-placed insight, or a gap card not reflecting a just-updated gap_score. Fixed by matching any key equal to or prefixed with table + '|' instead of an exact bare-table match.

2. Self-inflicted duplicate MutationObserver. Phylum Path's own auto-detect (shipped minutes earlier this same session) installed a second observer on #send-btn's disabled attribute, duplicating RPGACE.utils._initPhylaObserver's existing one and re-querying the chat DOM a second time on every single completed Oracle response — exactly the kind of duplicate-implementation pattern this project has repeatedly found and fixed before (three independent phyla-keyword lists, three chat-parsers). Fixed by having _runPhylaScan fire a new shared hook ('oracle:response-scanned', via the existing RPGACE.hooks pub/sub already used for rpgace:ready/page:show) right after computing its per-phylum matches array, and having phylumPath subscribe to that instead of owning its own observer — also means it now reads phylum 1's score straight out of the already-computed scan instead of re-scoring the text a second time. Down from 4 MutationObserver instances to 3; makes any future per-phylum auto-detect a hook subscription, not another observer.

STEP F8 — Formal phylum-selection criteria SHIPPED JULY 12
Replaced the blunt raw-count 2+ keyword threshold with a weighted scoring model, applied everywhere phylum matching happens. Concretely:

1. Full 21-phylum coverage — 7 phyla (11, 15, 17–21) had zero keywords and could never be silently matched by anything; all now have real keyword sets. 21 (Miscellaneous) deliberately kept empty — it's the catch-all bucket, giving it keywords would let the scanner match on it when it shouldn't.

Weighted terms — every keyword is now {term, weight}: 2 for terms specific/rare outside their phylum (sidechain, beatstars, 808), 1 for generic terms that could plausibly show up in several (style, tone). Architecture fragility — ~25 places in rpgace_core.js may share the rpgace:ready re-subscription bug FLAGGED, NOT AUDITED
Found while diagnosing why the Phase 2 nav tab wasn't appearing (see the July 16 card above): RPGACE.register() wires every module's init() as a listener on 'rpgace:ready' itself, so init() only ever runs because that event already fired — but RPGACE.hooks.fire()'s plain Array.forEach never revisits listeners pushed onto the array during its own iteration. Any module that calls RPGACE.hooks.on('rpgace:ready', ...) from inside its own init() is registering a listener too late for the pass currently invoking it — silently, no error, forever. phylumPath's 2 occurrences (side-panel button, nav tab) were fixed. Grep found ~25 more occurrences of the same call in rpgace_core.js beyond R.register's own definition — each needs checking individually: called from inside that same module's own init() (the buggy pattern) versus genuinely reacting to another module's readiness from elsewhere (fine). Not audited this pass.

Purpose
This is a living, dated log — not a replacement for the V6 architecture manual. Each session that ships fixes or features gets its own dated entry, appended below the most recent one, newest at top. It exists specifically to prevent re-diagnosing the same bug twice and to keep an honest record of what was tried and failed, not just what eventually worked.

Format for future entries: Summary → Shipped → Regression/Incident log (if any, kept in chronological order including wrong turns) → Root cause detail (for anything that took more than 2 exchanges to diagnose) → Lessons logged → Still open. Every entry gets a date-tag and its own anchor in the sidebar nav.
