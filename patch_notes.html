⚡ July 14 — efficiency audit: 1 real correctness bug + 1 self-inflicted duplication, both fixed
Asked to audit the codebase for efficiency/speed/workload upgrades after Phylum Path shipped. Found two, both fixed (Council of 5 verdict on each: 9/10 — concrete, low-risk, verified before reporting):

1. Cache-busting on write has never actually worked. RPGACE.cache (the 60s in-memory layer transparently wrapping every RPGACE.sb.select() call) stores entries under composite keys (table + '|' + params), but RPGACE.sb.insert()/update()/del()'s cache-busting wrappers all call RPGACE.cache.clear(table) with just the bare table name — which never matched a real stored key, so delete this._store[key] was silently deleting nothing. Confirmed via a standalone simulation before and after the fix. Real impact: any re-fetch of a cached table (taxonomy_tree, taxonomy_nodes, etc. — everything except the explicit noCache list) within 60 seconds of a write could silently show stale data — e.g. Phylum Path's tree view not showing a just-placed insight, or a gap card not reflecting a just-updated gap_score. Fixed by matching any key equal to or prefixed with table + '|' instead of an exact bare-table match.

2. Self-inflicted duplicate MutationObserver. Phylum Path's own auto-detect (shipped minutes earlier this same session) installed a second observer on #send-btn's disabled attribute, duplicating RPGACE.utils._initPhylaObserver's existing one and re-querying the chat DOM a second time on every single completed Oracle response — exactly the kind of duplicate-implementation pattern this project has repeatedly found and fixed before (three independent phyla-keyword lists, three chat-parsers). Fixed by having _runPhylaScan fire a new shared hook ('oracle:response-scanned', via the existing RPGACE.hooks pub/sub already used for rpgace:ready/page:show) right after computing its per-phylum matches array, and having phylumPath subscribe to that instead of owning its own observer — also means it now reads phylum 1's score straight out of the already-computed scan instead of re-scoring the text a second time. Down from 4 MutationObserver instances to 3; makes any future per-phylum auto-detect a hook subscription, not another observer.
🧬 July 14 — "Phylum Path" shipped: bottom-up insight → article pipeline (Phylum 1 pilot)
New feature, distinct from the existing top-down taxonomyTree.proposeLineage/silentPropose (which places one topic into a path from a phylum down to a leaf, in one shot). Phylum Path instead starts from a granular teaching insight and builds up: the insight attaches at whatever rank it genuinely needs (Order through Variant, flexible depth), and any rank along the way can get its own synthesized reference article. Scoped via an 8-question questionnaire (full record in /root/.claude/plans/woolly-watching-lamport.md) before any code was written — decisions: both auto-detect and manual entry, depth flexes per insight, new insights only (no retroactive repair of Phylum 1's pre-existing 23 flat-parent_id nodes), piloted on Phylum 1 (Compositio) only, article regeneration is manual-button-only, every rank gets its own article including the phylum root, and the persona extends Prod Oracle's existing 3-layer method ("simple terms → technical mechanics → expert nuance") rather than a new prompt shape.

New phylumPath module: "🧬 Phylum Path" button in the Oracle quick-row opens a panel with a manual insight textarea, a live tree view of Phylum 1's current structure (each node showing its rank name via the new taxonomyTree.RANK_NAMES/rankNameForDepth() — the first live-app use of rank naming, previously a taxonomy_map.html-only display convention), and a "📄 Generate/Refresh Article" button per node plus one for the phylum root itself. _placeInsight() fetches the existing tree, asks Oracle to decide an attach point (exact-matching an existing path, extending _checkForMorph's exact-match idea) and however many new ranks are genuinely needed — reasoned through 5 explicit checks (pedagogical clarity, non-redundancy, practical applicability, structural fit, expansion headroom), the phylum-specific operationalization of this project's Council-of-5 convention as a real prompt instruction rather than only a human planning ritual. New steps insert via the same chained-parent_id pattern as _acceptLineage. _generateArticle() gathers a node + all its descendants' explainer/deep_content, synthesizes via Oracle, and saves through the existing saveOracleToEncyclopedia() + taxonomy_node_id linking F7 already established — no new article storage invented.

Auto-detect entry point: a lightweight scan (reusing the same #send-btn completion signal RPGACE.utils._initPhylaObserver already watches) scores every completed Oracle response against Phylum 1's newly-enriched keyword list; above threshold, a small "🧬 Add to Phylum Path?" button appears under that message — opt-in, not auto-committing, consistent with how F4/F5/F6 also queue for review rather than silently writing.

Real data finding that shaped the scope: live-queried Phylum 1's actual 23 taxonomy_tree rows before designing this — all have parent_id: null (the known pre-July-12 flat-lineage bug, confirmed still un-migrated), and deep_content is empty on every single node despite _generateNodeContent supposedly populating it (a second, previously-undiscovered bug — not investigated, per the questionnaire's answer; Phylum Path uses its own separate content-gen call regardless). One existing leaf's name field is also a ~600-character block of unrelated quest text, a live example of the "unusually long name" data-quality flag taxonomy_map.html's own QA already checks for.

Not yet hand-tested. No live browser session available in this environment to click through it — needs a real pass placing an insight, confirming the attach/new-path decision looks sound, and confirming an article actually lands in Encyclopedia correctly linked.

STEP F8 — Formal phylum-selection criteria SHIPPED JULY 12
Replaced the blunt raw-count 2+ keyword threshold with a weighted scoring model, applied everywhere phylum matching happens. Concretely:

1. Full 21-phylum coverage — 7 phyla (11, 15, 17–21) had zero keywords and could never be silently matched by anything; all now have real keyword sets. 21 (Miscellaneous) deliberately kept empty — it's the catch-all bucket, giving it keywords would let the scanner match on it when it shouldn't.
2. Weighted terms — every keyword is now {term, weight}: 2 for terms specific/rare outside their phylum (sidechain, beatstars, 808), 1 for generic terms that could plausibly show up in several (style, tone). Threshold moved from raw count ≥2 to weighted score ≥3 (RPGACE.utils.PHYLA_MATCH_THRESHOLD), so two generic words alone can no longer trigger a match the way they used to.
3. Word-boundary matching — was naive text.includes(keyword), which false-positived on short terms inside unrelated words ("key" matching "monkey"/"turkey"/"keyword"). Now regex word-boundary matching.
4. Fixed a real overlap — 14 (Visio Cinematica) and the new 11 (Lingua Musicae) both claimed "colour" and "mood board". Split them: 14 keeps concrete filmmaking-production terms, 11 keeps abstract aesthetic/branding terms.
5. Found and consolidated a third independent copy of this same keyword list, in contentRepurpose._detectPhyla — missed by F2's consolidation, still on the old 14-phylum/raw-count model. How This Manual Works
Purpose
This is a living, dated log — not a replacement for the V6 architecture manual. Each session that ships fixes or features gets its own dated entry, appended below the most recent one, newest at top. It exists specifically to prevent re-diagnosing the same bug twice and to keep an honest record of what was tried and failed, not just what eventually worked.

Format for future entries: Summary → Shipped → Regression/Incident log (if any, kept in chronological order including wrong turns) → Root cause detail (for anything that took more than 2 exchanges to diagnose) → Lessons logged → Still open. Every entry gets a date-tag and its own anchor in the sidebar nav.
