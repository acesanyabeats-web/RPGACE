🏗️ July 18 — Bookworm's URL/PDF chapter detection rebuilt from scratch, per Alex's own architectural insight
After 4 separate fix attempts on the same missing-chapters bug (whitespace-fuzzy matching, gap-fill retry, bounded structural fallback) each improved things but the missing/misordered set kept shifting between runs — Alex diagnosed the real underlying pattern directly: the manual TOC-paste path has never failed once, across every single test, because it only ever asks the model to read a clean table of contents — it never asks the model to recall body text it may not have actually seen. The URL/PDF path's searchString mechanism did exactly that: asked the model to "quote verbatim" a chapter's opening line, which — confirmed directly via Vercel logs — it sometimes couldn't do, because that chapter's real content lay beyond its 25000-char viewing window (this book's own detailed table of contents alone fills that window). It would then reconstruct a plausible-sounding guess from that chapter's own TOC sub-headings, which correctly failed to string-match anywhere real, because it was never real.

Rebuilt api/bookworm-fetch.js around Alex's proposed architecture — two clean, independent steps instead of one fragile one:
1. detectChapterListByOracle() — the model's ONLY job is reading the table of contents: chapter number, title, page number if shown, keywords, suggested phylum. It is never asked to recall or quote any body text at all — the same scope the manual-paste path has always used, and the same reason that path has never once failed.
2. resolveChapterHeadingsMechanically() — fully deterministic, zero model involvement. Every chapter's number+title pattern is searched for across the WHOLE extracted text; the tightly-clustered front-matter TOC mentions are stripped (same declustering principle as the existing dropClusteredBoundaries); then chapters are walked in reading order, each one assigned the first remaining occurrence after the previous chapter's position. This naturally lands on the real heading, skips the TOC decoy, and skips any later cross-reference (a chapter is essentially never referenced by number before it's actually reached in a book) — solving the ordering bug from the last fix pass as a side effect of the design, not a bolt-on.

Net effect: the URL/PDF path's chapter detection now works on exactly the same principle that's made the manual path 100% reliable every time it's been tested — read a list, don't recall unread prose. The gap-fill retry mechanism (2 prior sessions' worth of patches) is removed entirely; it's structurally unnecessary now, since detection no longer depends on the model having "read" anything beyond the TOC.

Honest flag: not yet hand-tested — this is a real rebuild, not an incremental patch, and deserves a fresh PDF-upload test before being trusted. Page-number-assisted extraction (using the TOC's own page numbers to narrow the PDF.js per-page search, part of Alex's original ask) was deliberately scoped OUT of this pass — mapping a book's printed page numbers to actual PDF file page indices requires inferring a front-matter offset, a real source of risk not worth taking blind in the same pass as this rebuild. Logged as a genuine follow-up, not dropped silently.

🎯 July 18 — Real root cause found directly, no diagnostic round needed: PDF text loses spaces at word boundaries
Alex pasted the actual raw extracted table-of-contents text directly, and it showed the answer immediately — no need to wait for the logging added last round. The real heading text has zero space in places a human reader sees one: Chapter14, AdditiveRhythms, MusicalSound. Not a bug specific to this book or a PDF.js quirk — a genuine PDF-encoding pattern (many PDFs position glyphs precisely for a stylized heading instead of using an actual space character), and it clearly varies page-by-page within the same book, which is exactly why some chapters resolved fine and others didn't: resolveChapterHeadingsMechanically's regex required real whitespace (chapter\s+14) between "chapter" and the number and between title words, so anywhere the real text joined them with zero space, it could never match, regardless of chapter number.

Fixed: both regex patterns changed from requiring whitespace (\s+) to tolerating zero-or-more (\s*), and the number's trailing \b boundary replaced with (?!\d) — a plain word boundary fails when a letter follows a digit with no separator at all ("Chapter14Additive..."), since both sides count as "word" characters to \b; (?!\d) only refuses a match if ANOTHER digit follows (guarding against "14" false-matching inside "140"), while correctly allowing a letter to follow immediately.

Also strengthened detectChapterListByOracle's prompt with an explicit self-verification pass before it answers — 5 concrete checks (completeness/no gaps, title accuracy, correct order, no duplication, page numbers captured) mirroring this project's Council-of-5 rigor, applied concretely to this one task rather than as theater — plus an explicit heads-up in the prompt itself that PDF-extracted text can lose spaces this way, so the model doesn't misread a joined word as a different chapter title.

Scope note: every test so far has used the PDF-upload path specifically (PDF.js extraction) — Jina (the separate "📖 Start" URL-fetch path) has not been exercised at all today. This fix applies to both since they share the same detection functions, but Jina's own text-spacing behavior is unverified without a live URL test.

🐛 July 18 — Real bug found AFTER the "finish line": chapter 1's actual body text was its own table of contents, invisible until someone actually read it

Root cause: every chapter after the first is protected from anchoring on its own table-of-contents mention because resolveChapterHeadingsMechanically requires its match to land after the PREVIOUS chapter's already-resolved position. Chapter 1 has no previous chapter — its search floor starts wide open — so the very first "Chapter 1 ... title" occurrence in the whole document gets picked, which is almost always the TOC listing itself, since by definition the TOC comes before the real content. Fixed: once the SECOND chapter is resolved, the first chapter's offset is re-derived as the LAST candidate match that still falls before the second chapter's real position — bounded so it can never overshoot into chapter 2, correctly preferring the later (real) occurrence over the earliest (TOC) one. Same "prefer later, but bounded" principle already used everywhere else in this pipeline, just finally applied to the one chapter structurally exempt from the normal cursor protection.

First real usage (pasting the actual "Music Theory for Computer Musicians" PDF URL) surfaced a real defect within minutes, confirmed live in Supabase: both attempts created a bookworm_books row with 0 chapters, both silently flipped to status: 'complete' the moment the book was opened.

Root cause: RPGACE.sb.insert() sends Prefer: return=minimal and never parses the response as JSON — it returns the raw, unparsed fetch() Response, not the inserted row. _startBook() used it expecting book.id back, so book.id was always undefined — every chapter row got inserted with book_id: undefined, failing the NOT NULL constraint server-side, while the client never checked the response status and just moved on as if it had succeeded. Architecture fragility — ~25 places in rpgace_core.js may share the rpgace:ready re-subscription bug FLAGGED, NOT AUDITED
Found while diagnosing why the Phase 2 nav tab wasn't appearing (see the July 16 card above): RPGACE.register() wires every module's init() as a listener on 'rpgace:ready' itself, so init() only ever runs because that event already fired — but RPGACE.hooks.fire()'s plain Array.forEach never revisits listeners pushed onto the array during its own iteration. Any module that calls RPGACE.hooks.on('rpgace:ready', ...) from inside its own init() is registering a listener too late for the pass currently invoking it — silently, no error, forever. phylumPath's 2 occurrences (side-panel button, nav tab) were fixed. Grep found ~25 more occurrences of the same call in rpgace_core.js beyond R.register's own definition — each needs checking individually: called from inside that same module's own init() (the buggy pattern) versus genuinely reacting to another module's readiness from elsewhere (fine). Not audited this pass.
