The pattern behind today's hardest stretch — six iterations on one calendar view

What actually went wrong, precisely
The Daily Grid compaction feature took six live-deploy rounds to get right. Looking back at the actual sequence, every failed round shared one root cause: positioning/layout logic was written and shipped before ever seeing the real rendered output. Absolute-positioning math was built on assumptions about row height, container offset, and content fit — each assumption plausible, each one individually wrong in a different way, and each wrong assumption only surfaced after a live deploy and a screenshot round-trip.

The fix that actually worked wasn't a smarter patch — it was abandoning the positioning-math model entirely for a normal-document-flow model (build rows in order, let the browser handle layout) that had no pixel math to get wrong in the first place. The lesson isn't "measure more carefully" — it's "prefer the architecture that has fewer ways to be wrong," even if it seems like more code up front.

Every specific bug this session traced to the same underlying cause

  • UTC vs local date parsing (3 occurrences: _calDateStr, openSchedModal, renderDailyGrid) — the same one-line mistake (toISOString() instead of local date components), copy-pasted or independently reinvented in multiple places across different build sessions
  • Two parallel scheduling systems (rpgace_scheduled_agendas vs rpgace_sched_agendas) — built at different times, never reconciled, silently incompatible for an unknown number of sessions before today
  • Duplicate button injection — one new inline button system built without checking whether an older system (_addSchedButtons) was already doing the same job from a different call site
Every one of these is a version of the same root issue: code was added without first checking what already existed and calling it. Not a knowledge failure — a verification-order failure.

New rules for future development — specific and checkable, not vague

  1. Before writing any layout/positioning code, get a screenshot of the current unmodified state first. Never write CSS/positioning logic based on inference about what a container "probably" looks like — pull the real computed styles or a screenshot before writing math against them.
  2. Before adding a new UI behaviour (buttons, click handlers, injected content), grep for existing functions that already touch the same DOM elements. "Does this already exist somewhere else" is now a mandatory check before writing new code, not an afterthought when something breaks.
  3. Prefer normal document flow over absolute positioning whenever the two produce the same visual result. Absolute positioning has more ways to silently drift (offsets, measured heights, container padding) — only reach for it when flow genuinely can't do the job.
  4. When a fix doesn't work on the first try, stop and ask for the real rendered output (screenshot) before attempting a second blind patch. Today's worst stretch happened specifically when patches were applied back-to-back without a visual check in between — the correct discipline (verify before extending) was already known from last session's Oracle debugging, and got abandoned under momentum tonight.
  5. Big multi-part feature requests get a full spec-and-defer pass before any code, not a partial build. Schedule Oracle's five-subsystem scope was correctly deferred whole, rather than half-built and left inconsistent — this should be the default response to any request stacking 3+ genuinely new subsystems in one ask.

What this session confirms about the working relationship, worth stating plainly

Direct pushback when something wasn't working ("really shit job," "why cant you just delete the old version") led to faster, more precise fixes than continued hedging would have — the correction each time was specific enough to act on immediately. That pattern is worth preserving deliberately going forward: blunt, specific feedback beats polite ambiguity for this kind of iterative build work.

Judgment calls worth stating plainly

1. The four architecture changes were not wrong in isolation — they were premature
Streaming Oracle, a Supabase cache layer, and a tighter MutationObserver are all reasonable things to eventually want. The mistake was building all four at once, off a general performance trace, before isolating what the user was actually complaining about. A trace showing "Oracle calls take 5-7s" and "encyclopedia queried 3x" doesn't mean either one is *the* problem the user experienced — it means those are two things that showed up in a trace. Only re-recording a trace of the exact complained-about action (login → lag → unlag) surfaced the real cause.
2. Console overrides were repeatedly mistaken for fixes
RPGACE.streamOracle = null typed into console appeared to fix Oracle twice tonight, and both times it was diagnostic evidence, not a working fix, because it didn't survive a refresh. This cost at least two extra rounds of back-and-forth before the actual fix (moving the null-assignment to module init() instead of the rpgace:ready hook) was found.
3. Toast/success messages were trusted over re-querying actual state
A Supabase DELETE returned "Success. No rows returned" and was treated as confirmation of full completion. It had only partially run. This was caught only because the next step (adding a UNIQUE constraint) failed loudly with a specific duplicate title in the error message — if that next step hadn't existed, the partial DELETE could have gone unnoticed indefinitely.
4. The real root cause was one layer deeper than either of us initially looked
Client-side (localStorage) dedup logic was correct and was still failing to hold, which looked like a client-side bug. It wasn't — the actual duplicate-producing writer was a local Python server (port 7842) inserting directly into Supabase with no existence check, something neither browser console nor client-side code could ever see or fix. The lesson: when a fix that should work keeps not holding, the write is probably happening somewhere the current investigation isn't looking, not that the fix logic is wrong.

Two real bugs found + fixed: shift date off-by-one, schedule key mismatch FIXED — JULY 6
Bug 1 — shifts displayed one day early on Weekly calendar: _calDateStr() used toISOString().split('T')[0], which converts to UTC before formatting. _calMondayOf() builds each day using local-time methods (setHours(0,0,0,0), getDay()) — so in BST (UTC+1), local midnight becomes 23:00 the previous day in UTC, and the date string used to look up shifts didn't match the date actually being displayed. Fixed by replacing toISOString() with a local-date-component formatter (getFullYear()/getMonth()/getDate()) that never converts timezone.

Root Cause Analysis
The Real Lag: Encyclopedia Duplication
Oracle being broken was a real bug, but it was not the cause of the original "site is lagging" complaint. This is what actually was.
Diagnostic method that worked
Chrome DevTools → Performance tab → record login → stop after unlag. Flame graph showed one ~31,000ms red Task block, entirely inside refreshEncyclopediaDisplay → renderEncEntries → sortEntries, called three times in immediate succession.

The chain of evidence

CheckResult
localStorage entries112 (should have been ~38)
Unique titles in localStorage38 of 112 — 74 duplicates
Supabase encyclopedia table104 rows, 38 unique titles
Worst offender"The Trending Artist Tutorial Format..." — 5 copies
Client-side dedup logic (encSync)Correct, but only cleans localStorage 2.5s after sync — cosmetic, not preventative
Actual insert sourcepushLocalToSupabase() in frozen main.js, called via localhost:7842/push-to-supabase — a local Python server, not the browser

Why client-side fixes kept failing

Every localStorage cleanup was overwritten on the next sync because Supabase itself held the duplicates. Cleaning the client is treating a symptom — the real writer was a local server process (port 7842, likely local_server.py or rpgace_intel.py) inserting rows with no existence check, at some point earlier in past sessions while that server was running.

The permanent fix

-- Step 1: remove existing duplicates, keep lowest id per title
DELETE FROM encyclopedia a
USING encyclopedia b
WHERE a.title = b.title
  AND a.id > b.id;

-- Step 2: make duplicate titles structurally impossible
ALTER TABLE encyclopedia
ADD CONSTRAINT encyclopedia_title_unique UNIQUE (title);

First DELETE attempt (using ROW_NUMBER() OVER (PARTITION BY title ORDER BY created_at)) partially failed — 5 titles retained 2 copies each, likely due to a created_at tie or NULL breaking the ordering. Second attempt used id as the tiebreaker instead — always unique, never NULL — and succeeded cleanly.

Confirmed result
104 rows → 38 rows in Supabase. localStorage re-synced to match. Performance re-recorded after hard refresh: the 31,000ms scripting block is gone. User confirmed: "it works way quicker now."

Why this matters going forward

The UNIQUE constraint means no future insert — from any script, any session, any button — can ever create a duplicate title again. This closes the bug at the only layer that actually controls it. It also makes auto-sync safe to consider in a future session, since duplicate insertion is now impossible regardless of how many times sync fires.

Recursive Taxonomy Tree — full architecture CONFIRMED — MAJOR BUILD — NEXT SESSION
Built via GODMODE + Council of 5, July 2 evening. This is the single largest architectural addition since the original 21-phylum taxonomy — replaces the current flat Phylum→concept structure with genuine unlimited-depth drill-down, and adds an unattended proposal-and-review pipeline so the tree grows passively from every content source, not just live Oracle chats.

Core decision — one self-referencing table, not fixed levels: biological taxonomy's fixed ranks (Class/Order/Family/Genus) were rejected because depth genuinely varies per branch — some topics resolve in 3 steps, others need 21. A single taxonomy_tree table with a parent_id self-reference supports any depth without schema changes.

Two new tables:
  • taxonomy_tree — the live, accepted tree.   • taxonomy_proposals — a staging table. Nothing writes to the live tree until explicitly accepted. Holds source_type (content_intelligence/encyclopedia/oracle/manual), the proposed full path, proposed steps as JSON, and a status field.

  • Reject — discarded, no tree changes

Fix: one RPGACE.utils.getRecentOracleMessages(n) helper added to the config module. All three existing callers switch to use it instead of their own DOM-walk. Same output, one shared implementation. Any future module needing "what did Oracle just say" gets it for free instead of writing a fourth parser.

Bonus fix this unlocks: once centralised, this helper can be pointed at conid_pot (Supabase) as a secondary/preferred source instead of raw chat DOM — which would resolve the earlier-flagged bug where the Repurpose dropdown sometimes grabs the wrong idea from a long Oracle response, since ConIDPot entries are already clean, titled, and structured, unlike raw chat text.
Timestamped Insights + Self-Correcting Frame Pull SCHEDULED
Final scoped version, replacing two earlier drafts (manual-only attach, and full auto-extraction — both superseded). Confirmed buildable: yt-dlp is already integrated in RPGACE, FFmpeg is already installed locally, Whisper transcripts are already timestamped per segment.

How it works:
  1. When Content Intelligence generates an insight from a video, tag it with the Whisper segment timestamp it came from (e.g. "0:38–0:46") — one field added to the existing pipeline, no new pipeline stage
  2. A "Pull screenshot" button next to each insight triggers yt-dlp (already integrated) + FFmpeg to grab 3 candidate frames spaced across that segment window, not one guessed frame — on demand, only for insights you actually click, not all frames in the video
  3. You pick the frame that actually matches the insight, or reject all three — this is the confirm/reject step, and it doubles as free labeling data with zero extra manual effort since you were reviewing the insight anyway
  4. Each confirmed pick logs its offset from segment-start. After ~15-20 confirmed insights, the system has a real running average of where in a segment the useful frame tends to sit for your specific tutorial-watching footage style — the 3 candidate frames start biasing toward that learned offset instead of a generic even spread across the segment
What this is not: not a trained vision model, not FL Studio element recognition, not frame-by-frame extraction of full videos. It's a timestamp field, an on-demand 3-frame pull, and a running offset correction — cheap, buildable now, and it is the version where today's manual work genuinely improves tomorrow's automated pull, which none of the earlier drafts actually achieved.

Known remaining limitation: Whisper segment precision varies by video — some segments span 8+ seconds. The 3-candidate spread exists specifically to absorb this imprecision rather than assume timestamp accuracy.

Purpose
This is a living, dated log — not a replacement for the V6 architecture manual. Each session that ships fixes or features gets its own dated entry, appended below the most recent one, newest at top. It exists specifically to prevent re-diagnosing the same bug twice and to keep an honest record of what was tried and failed, not just what eventually worked.

Format for future entries: Summary → Shipped → Regression/Incident log (if any, kept in chronological order including wrong turns) → Root cause detail (for anything that took more than 2 exchanges to diagnose) → Lessons logged → Still open. Every entry gets a date-tag and its own anchor in the sidebar nav.
