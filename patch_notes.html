🐢 July 22 — Real root cause found for "left nav doesn't respond for ~10s after login": onReady() was waiting for the wrong signal, not just for leftNav specifically
Alex's report: page loads, dashboard scrolls fine, but clicking anything in the left list does nothing for the first ~10 seconds. GODMODE evidence pass instead of another guess.

Real root cause: onReady() — the function every one of ~40 modules' init() calls wait behind via rpgace:ready — gated on document.readyState==='complete' (the window 'load' event), which waits for EVERY subresource on the page (every image, every font) to finish downloading, not just the DOM. The comment justifying this said "give initApp() time to run" — checked checkPassword() in main.js directly: initApp() is called from the login handler, gated entirely by WHEN Alex logs in, completely decoupled from page-load timing. It can run long after window.load already fired, or need to run before it ever will. There was never a real dependency on waiting for images/fonts — just unnecessary latency stacked in front of every module's startup, worst-case as long as the slowest image/font on the page takes.

Fix: onReady() now checks document.readyState !== 'loading' (DOMContentLoaded-equivalent — fires once the DOM itself is parsed, which it already is by the time rpgace_core.js runs, since both scripts load at the very end of <body>) instead of waiting for full 'complete', cutting the delay from window.load + 150ms to effectively DOMContentLoaded + 50ms. This is a systemic fix — every module's startup timing improves, not just leftNav's.

Second, separate bug found in the same pass: leftNav._go(page) unconditionally dereferenced RPGACE.CONFIG.pages.phylumPath on every single click, regardless of which page was clicked. If RPGACE.CONFIG ever wasn't ready yet at click time, this threw before ever calling showPage() — the click would silently do nothing, no error shown, no toast, exactly matching Alex's description. Guarded defensively, same pattern as _items()'s existing CONFIG guard.

Stale cache-busting version found again — the version string was bumped once already today (?v=20260722a) after the earlier incident, but rpgace_core.js changed again after that bump for this fix, so it was bumped a second time to ?v=20260722b. Repeating the lesson from earlier today: bump this every time rpgace_core.js changes, not just once per day.
