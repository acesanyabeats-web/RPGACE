IDEA 1 — OPENMONTAGE (github.com/calesthio/OpenMontage)
================================================================
Alex's own framing (paraphrased from the pasted repo README + his
message): "this would hugely help video pipeline for video creation, it
could also work with visual oracle and beat log to sync with beat."

WHAT IT IS (from the pasted README, not assumed): an open-source,
agent-operated video production system - 12 pipelines (explainer,
cinematic, documentary montage, etc.), 100+ Python tools, 700+ skill
files. Works by being cloned as its own repo, driven directly by an AI
coding assistant (Claude Code, Cursor, etc.) via YAML pipeline manifests
+ Markdown stage-director skills. Requires Python 3.10+, FFmpeg, Node
18+, its own venv, and a Node/npm `remotion-composer` subproject. Has a
genuine zero-API-key mode (Piper TTS, free stock/archive footage,
Remotion composition) plus paid providers for AI-generated visuals/video.

REAL EVIDENCE GATHERED (GODMODE, before judging):
- RPGACE's own F16 (Beatstars listing)/F17 (video pipeline stages,
  `video_jobs` table)/F18 (auto visual treatment) are real but NEVER
  hand-tested (still in patch_notes.html's Still Open section) and
  produce TEXT only (listing copy, visual-treatment suggestions) - no
  confirmed working video-rendering engine exists in RPGACE today. Real,
  concrete capability gap.
- CORRECTION (Alex caught this, July 24, re-verified via real repo
  search): the first pass of this judgment overstated the toolchain
  mismatch as "Python+Node+FFmpeg vs. RPGACE's zero-build vanilla JS."
  Real evidence: Python genuinely IS part of RPGACE (`rpgace_build.py` +
  ~100 one-off patch scripts in `scripts/`, this project's whole edit
  history has used ad hoc Python); Node.js genuinely IS part of RPGACE
  (`package.json` exists specifically because Vercel's `api/*.js`
  functions run on a real Node.js runtime - confirmed via deployment
  data showing `lambdaRuntimeStats: {"nodejs":11/12}`). FFmpeg alone is
  genuinely absent (zero references anywhere in the repo). The real,
  precise distinction is narrower than first stated: OpenMontage needs a
  FORMALLY dependency-managed Python project (`requirements.txt`, its own
  venv) and a real npm-managed Node subproject (`remotion-composer`) plus
  the FFmpeg binary - a different SHAPE of usage (package-managed,
  standalone) than RPGACE's own ad hoc no-package-manager scripts and
  Vercel-only Node runtime, not an outright absence of the languages
  themselves.

/DEBATE:
Team FOR: a genuine, confirmed capability hole exists (no real video
rendering in RPGACE today). OpenMontage is mature (41.7k stars, active),
free-tier-capable, and agent-operated by design - a Claude Code session
could run it directly, TODAY, as a separate side-tool, with the finished
.mp4 flowing back into RPGACE's Content Production/F17 tracking as a
completed asset. Matches the governing 48-hour rule directly if Alex is
making real promotional/beat videos now.

Team AGAINST (real, not a strawman): standing up a second heavyweight
toolchain is real ongoing maintenance overhead for a project whose
explicit principle is minimalism/zero-build-step. Beat-syncing (BPM/key/
mood from RPGACE's Beat Log flowing into an OpenMontage brief) is NOT
automatic - it needs a real bridging step that doesn't exist and wasn't
specified. Cloning and operating a third-party 700+-skill-file repo hands
a coding agent broad read/execute scope and, with paid providers
configured, real API keys - non-trivial even for a reputable public repo.

VERDICT: Adopt as a SEPARATE OPERATED TOOL, NOT an embedded RPGACE
integration. Real adoption path (unlike Precedents 1/2's outright
rejections): operate OpenMontage in its own directory/environment
whenever Alex wants an actual rendered video from a Beat Log entry or
Visual Oracle brief; the finished asset gets logged into RPGACE's own
F17/Content Production tracking by hand (or via a small future glue
script) - never embedded into RPGACE's own Vercel runtime. Beat/Visual-
Oracle sync is a real FUTURE enhancement, not automatic out of the box -
stated honestly so it's never assumed working before it's built.

IDEA 2 — "CAN RPGACE USE CLAUDE CODE THROUGH SUPABASE WHEN API TOKENS
ARE COMPLETELY OUT?"
================================================================
Alex's own framing (verbatim): "Can rpgace also use Claude code through
supabase when API tokens are low out completely out? would be an awesome
update, possibly a river."

REAL EVIDENCE GATHERED:
- Claude Code Remote's trigger infrastructure (`create_trigger`,
  `fire_trigger`) already has a real, WORKING precedent in this exact
  project: the daily Morning Brief Routine (`trig_015uRjGVrdmiwc5zxq47mMA6`,
  built July 22) - curls Supabase directly, writes back into `journal`.
  This is the exact mechanism this idea would reuse, not new
  infrastructure.
- `create_trigger`'s own real constraint: minimum interval is HOURLY.
  A fallback lane built this way is asynchronous/batch, not synchronous.

/DEBATE:
Team FOR: directly reuses an already-proven pattern in this exact project
- zero new infrastructure concepts, just a new Supabase queue table and a
second Routine. Solves a real, recurring, already-documented pain
(Anthropic API credits running out mid-project, more than once this
session alone) by giving async/background AI jobs (taxonomy placement,
Bookworm chapter analysis, Content Intelligence proposals - all already
async by design) a second, completely separate quota lane to fall back
to when the app's own key is dry.

Team AGAINST: the hourly-minimum constraint means this is fundamentally a
BATCH fallback, not a live one - it cannot make Oracle's interactive chat
"still work instantly" when credits run out, and presenting it that way
would be dishonest. It's also a real, if small, new moving part (a queue
table, a Routine, RPGACE-side polling/UI changes to show "queued" instead
of an instant reply) - not zero-cost, and only pays off exactly when the
primary API is already down.

VERDICT: Adopt WITH MODIFICATION - scope it explicitly as an ASYNC
FALLBACK LANE for background AI jobs only (Bookworm/Content-Intelligence/
taxonomy placement), never presented as a live-chat replacement. Real,
buildable, reuses proven infrastructure. Not yet built - a real future
session's work once Alex confirms he wants it.
