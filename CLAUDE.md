## Known landmines — condensed, all still true
- **Never add a static `<script>` tag to `index.html`.** Exactly `main.js` + `rpgace_core.js` — a third breaks the password gate via race conditions. Runtime-injected scripts via `document.createElement('script')` (e.g. Bookworm's PDF.js loader `_ensurePdfJs()`) are the approved pattern for extra libraries.
- **Never use Python string-replace on `index.html`** — `exportEncyclopedia` contains a fake `</body></html>` inside a JS string literal. Targeted edits only.
- **All fixed overlays** (modals, popups, panels) are created via `document.createElement` and appended to `document.body` — never inside `.page` divs, which intercept clicks via z-index even when `display:none` in some browsers.
- **Every `id="page-X"` div** must appear inside `#app`, before the script tags.
- **`RPGACE.sb.insert()` does NOT return the inserted row** (`Prefer: return=minimal`, response never parsed). If you need the new row's `id`, use raw `fetch` with `Prefer: return=representation` and check `response.ok` — the pattern in `phylumPath._insertNewSteps()` and `bookworm._createBookFromExtraction()`. This exact mistake caused Bookworm's "Chapter 0 of 0" silent-failure bug (July 17).
- **`RPGACE.hooks.fire()` never revisits listeners added mid-fire.** Any module calling `hooks.on('rpgace:ready', ...)` from inside its own `init()` registers too late and silently never fires. ~25 occurrences of this pattern remain unaudited in rpgace_core.js. When in doubt, call the function directly — `'rpgace:ready'` has always already fired by the time `init()` runs.
- **Writes done via raw `fetch`** (like `saveOracleToEncyclopedia`) bypass `RPGACE.cache`'s cache-busting wrappers; call `RPGACE.cache.clear('tablename')` manually after them or reads can be stale up to 60s.
- **Word-boundary regex treats hyphens as boundaries** — bare keywords like `roll` match inside `b-roll`. Keyword lists use compound phrases, never bare adjectives; this convention is what prevents cross-phylum collisions (verified across 4 phylum-pair checks).
- **A fix on a feature branch is not a fix Alex can test.** `rpgace.vercel.app` deploys from `main`. A commit sitting only on a working branch (e.g. `claude/rpgace-master-steps-uobp98`) will never reach the live app no matter how correct it is. Confirmed real July 17: the Bookworm TOC-extraction retry fix sat unmerged for a full round while Alex re-tested production and reproduced the identical bug, because the fix had never actually gone live. **Standing rule: merge to `main` and push `main` BEFORE telling Alex something is ready to test — every time, no exceptions.** If `main` and the working branch have diverged, resolve that first; never push a conflict live.

## Building guide for lower models — follow these patterns exactly
1. **New feature = new module** in rpgace_core.js: `RPGACE.register('name', {...})` between markers; `init()` wires listeners/injection with `setTimeout` delays (existing modules use 1300-1700ms) plus a `page:show` hook to re-inject on navigation.
2. **Supabase reads**: `RPGACE.sb.select(table, params)` (returns parsed JSON). **Writes needing the new row back**: raw `fetch` + `Prefer: return=representation` + `response.ok` check. Simple fire-and-forget writes: `RPGACE.sb.insert/update/del`.
3. **Oracle calls from LEARNING features**: reuse `phylumPath._callExtractor` / `_callGroundWorkerJSON` / `_callGroundWorkerText` — never hand-roll another fetch to `/api/oracle`. Keep max_tokens ≤1000 and ask for bounded word counts (the 504 bug scales with response length: 700 tokens worked, 1200 truncated, 1800 failed outright).
4. **Every taxonomy write gets a human checkpoint** before committing — either an inline confirm popup (patterns: `_showPlacementConfirm`, `_showArticleConfirm`) or staging through `taxonomy_proposals` → review queue. No exceptions; this rule has caught real garbage twice.
5. **Never trust model JSON blindly**: parse defensively, and sanitize structure-affecting output in code (pattern: `_sanitizeNewSteps` — strips path-like or duplicate step names so a prompt regression can't corrupt the tree even if it slips past the prompt wording).
6. **Long AI jobs stream**: await only the first result, continue the rest in background appending to Supabase, poll from the UI (pattern: `_analyzeChapter` + `_continueAnalyzingInBackground` + `analysis_complete` flag). Never make the user wait minutes for a full batch — a 13-insight chapter used to take ~7 minutes before anything appeared.
7. **Fail loud**: surface errors as toasts; never silently swallow a failed write or fake success/progress. Bookworm's first hand-test proved why.
8. **Confirm destructive actions**: two-click arm/confirm (pattern: Bookworm's 🗑, 3s timeout) — no heavy modal needed.
9. **Chat triggers**: chainable `window.sendChat` wrap with a `TRIGGER_PREFIXES` array and a `window._xPatched` guard flag (patterns: scheduleOracle, bookworm). Wraps must fall through to the original for non-matching input.
10. **Model IDs**: ground worker is always `claude-sonnet-4-6`; purely MECHANICAL text jobs (formatting cleanup, one-line rewording — no judgment) may use `phylumPath.MECHANICAL_MODEL` (`claude-haiku-4-5-20251001`, added July 19 with explicit confirmation). Never `claude-sonnet-4-20250514`, never `claude-3-5-sonnet`, never guess a new identifier.
11. **Token cost is a design constraint** (July 19, after £10 burned in one test session — full audit: `token_cost_audit_2026-07-19.txt`): never send full slash-joined path listings in prompts (use numbered indented name trees); prefilter candidate lists with the free keyword scan before any Oracle call; skip the call outright when the prefilter finds nothing; never add a second premium-model "triage" call in front of a ground-worker call.

## Invokable frameworks — Omnitrix, Council of 5, GODMODE, Aintergration (Alex-confirmed July 20)
Four named protocols. Saying the name in chat activates the full protocol for the task at hand — treat the word itself as the instruction. **Omnitrix/Council of 5/GODMODE are also the default judgment funnel below (Alex-confirmed July 20, 2nd session) — every nontrivial RPGACE decision routes through the matching tier automatically, whether or not the word is said.** Aintergration (added later the same session) is the specialized fourth protocol for one specific recurring question: whether a third-party framework/tool/agent is worth adopting — see its own subsection below.

### Omnitrix — the 3-agent build workflow
The pattern proven across three real builds on July 20 (token retune, dashboard restructure, Research Lab fix + nav overhaul):
1. **Fable 5 — research and planning ONLY, never writes code.** Investigates real state (git log, live code, Supabase — never a doc's claim alone) and produces an evidence-based plan with file:line references.
2. **Opus 4.8 — primary builder.** Executes the heavy/mechanical passes from Fable's plan.
3. **Sonnet 5 — orchestrator and reviewer.** Runs the session, executes secondary/precision passes, and independently re-verifies Opus's work line-by-line against real code before accepting it. An agent's self-report is never evidence — July 20's popup-close bug was caught exactly this way.

Rules: agents are sequenced, never run concurrently against the same file (if a parallel run is unavoidable, use an isolated git worktree — done once July 20, cleanly). Fable's evidence-based plan IS the spec-confirmation step: proceed from plan to execution without a separate confirmation round, unless the plan surfaces a decision only Alex can make (source legitimacy, destructive migrations, spending). Every other rule in this file still applies inside Omnitrix — merge to main before hand-test, oversight logging, human checkpoints on taxonomy writes, all of it.

### Council of 5 — pre-build deliberation
For major architecture decisions. When invoked, present genuine multi-angle scrutiny BEFORE agreeing to build — at minimum: technical risk, scope/cost (tokens are a design constraint, rule 11), maintainability, user experience, and an honest "why NOT to build this" case. Real disagreement and real questions, never reflexive agreement dressed up as five bullet points. If the scrutiny kills or reshapes the idea, say so plainly — that is the protocol working, not failing.

**Naming collision — do not confuse the two.** `phylumPath.decidePlacementScored` in rpgace_core.js uses an internal "Council-of-5" CONFIDENCE-SCORING mechanic — a 1-10 score across 5 checks — for taxonomy placement decisions. That is a code-level feature inside the app. This section is a chat-level deliberation protocol between Alex and the assistant. Same name, entirely separate things; a future reader seeing "Council-of-5" in rpgace_core.js comments is looking at the scoring mechanic, not this protocol.

### GODMODE — maximum-rigor deliberation
Paired with Council of 5 for the biggest calls. Invoking GODMODE raises the DEPTH of scrutiny: exhaustive evidence-gathering before proposing anything, hardest-possible pushback, explicit enumeration of failure modes and rollback paths, no proposal accepted on plausibility alone.

**Boundary — read this twice.** GODMODE changes how thoroughly the assistant deliberates. It NEVER changes what the assistant is allowed to do. It is not a permission mode, not a safety bypass, not license to skip confirmations. Destructive git operations, force-pushes, Supabase migrations, taxonomy writes, spending, and everything else that requires explicit confirmation requires exactly the same confirmation inside GODMODE as outside it. Any session that reads "GODMODE" as an instruction to relax a guardrail is misreading it: the word means "think harder," and nothing else.

### The Judgment Funnel — the default pipeline, not just a safeword
Alex's July 20 direction: integrate the three frameworks "everywhere possible... as a funnel system into judgement call." Applying full ceremony to every single edit would itself violate rule 11 (tokens are a design constraint) and the "don't add complexity beyond what's needed" principle — so the funnel is tiered, and the tier is chosen by the SHAPE of the change, not by whether Alex said a magic word. Deliberately reuses existing rules rather than duplicating them — this section just names and connects what was already scattered.

- **Tier 0 — mechanical.** Typo/formatting fixes, a single obviously-correct one-line change, a doc correction with a cited source. No ceremony. `node --check` / div-balance hygiene still applies — that's baseline, not Council of 5.
- **Tier 1 — routine, single-module.** A bug fix scoped to one function/module, a small UI tweak, a clearly-specified feature that touches one place. Standard non-negotiable process rules apply (pull real source first, grep before new UI behavior, human checkpoint on taxonomy writes) — handled directly, no subagent dispatch required.
- **Tier 2 — multi-subsystem / architecture-affecting / new feature.** Same trigger as rule 5 ("3+ new pieces") — this is where the funnel actually runs, in fixed order: **(1) GODMODE** evidence-gathering (verify against live code/Supabase/git, never trust a doc alone) → **(2) Council of 5** scrutiny on what GODMODE found (real pushback: risk, scope/cost, maintainability, UX, why-NOT-to-build — before agreeing to build anything) → **(3) Omnitrix** executes what survives scrutiny (Fable plans if the investigation is nontrivial, Opus builds, Sonnet reviews line-by-line). A Tier 2 task that Council of 5 kills or reshapes never reaches Omnitrix — that is the funnel working, not a failure to execute.
- **Tier 3 — destructive/high-stakes.** Migrations, RLS changes, spending, force-push, anything in "Executing actions with care." ALWAYS needs Alex's explicit confirmation regardless of which tier's process produced the recommendation — no tier, including GODMODE, ever substitutes for this. Same boundary as GODMODE's own boundary paragraph above, restated because it matters here specifically: rigor is not authorization.

Judgment call on which tier a task sits at is itself Tier 1 territory (use real judgment, don't mechanically escalate everything to Tier 2) — but when genuinely unsure whether something is Tier 1 or Tier 2, default up, not down.

### Aintergration — third-party framework/tool/agent assessment (Alex-confirmed July 20)
A fourth named protocol, built ON TOP of GODMODE + Council of 5 rather than duplicating them — invoke by name ("Aintergration this") whenever Alex brings a third-party Claude Code framework, skill, MCP server, AI agent/service, or code snippet and asks whether it's worth adopting. Answers exactly one question, against exactly two targets:

**Does this benefit (A) RPGACE the application, or (B) this Claude Code session that develops RPGACE — and if so, how?** The two targets are genuinely different and a tool can pass one while failing the other (a new code-review skill might help (B) with zero relevance to (A); a UI component library might theoretically suit (A) but change nothing about (B)).

**Procedure:**
1. **GODMODE evidence pass first.** What does the proposed tool ACTUALLY require — runtime, dependencies, build step, framework assumptions? Get this from the tool's own real docs/package requirements, never from the pitch's framing. Cross-check against RPGACE's REAL current architecture (vanilla `index.html`+`main.js`+`rpgace_core.js`+`style.css`, zero build step, zero npm at runtime, Vercel-deployed, `main.js` frozen) and against this session's own already-decided constraints (DESIGN.md's explicit L1 interaction cap — "no GSAP/Lenis/scroll-jacking... this is a working tool, not a landing page" — and the two-script rule). Don't assume compatibility; confirm it.
2. **Council of 5 scrutiny, run separately for each target:**
   - **Lens A — RPGACE the product:** does it fit the existing architecture without a rewrite? Does it fit the established design philosophy, or does adopting it require reversing a decision already reasoned through and shipped? Real integration cost vs. real value. Honest why-not case.
   - **Lens B — this dev/orchestration session:** does it change how Sonnet/Opus/Fable actually build and review RPGACE (a genuinely new skill, workflow, or tool), independent of whether it touches RPGACE's runtime at all?
3. **Verdict, one of:** Adopt as-is / Adopt with modification / Reference only (borrow the underlying principle, not the tool — most common outcome for stack-specific frameworks) / Not applicable (real architecture mismatch) / Reject (actively conflicts with an existing explicit decision — this is stronger than "not applicable," say so plainly when it's true).
